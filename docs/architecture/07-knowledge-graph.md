## (a) Design rationale

The knowledge graph is not a separate product surface — it is the indexing structure that makes three downstream features possible:

1. **The Roadmap Planner** orders milestones by topological walk over `prerequisite_of` edges. Without an acyclic graph and a deterministic topo sort, the weekly plan is arbitrary.
2. **The Retriever's KG expansion** (from Deliverable 6) walks one hop along `related_to` and `example_of` edges when reranking returns a thin top-K. That single hop turns "I couldn't find anything" into "I found three related concepts that may help."
3. **The Course Workspace's interactive graph** renders Cytoscape JSON that the FE can lay out and animate without re-fetching. We pre-render the layout on the server when the graph is small (≤ 200 concepts); larger graphs ship without positions and the FE runs layout client-side.

Four constraints shape the implementation:

1. **The graph lives in Postgres, not Neo4j.** Two tables (`Concept`, `ConceptEdge`) from Deliverable 3 with proper indexes handle the entire workload — node count per course is hundreds to low thousands, edge count is single-digit multiples of that. Neo4j is an optional analytics mirror behind a feature flag (`kg.neo4j-mirror`), not a runtime dependency.
2. **Acyclicity is a contract, not an aspiration.** The Curriculum Builder agent must return an acyclic DAG, and the graph store validates that on write. A cycle reaches the database only through a `BYPASSRLS` admin operation, and the nightly verifier flags it.
3. **Topological order is deterministic.** Two roadmap regenerations over the same course must produce milestones in the same week-and-ordinal positions, or the student experience drifts. Tie-breaks are alphabetical by `concept.id` (UUIDs sort stably), never by insertion order.
4. **Cytoscape spec is the API output, not the storage shape.** The Postgres rows are normalised. The Cytoscape JSON is a denormalised projection generated by `graph/cytoscape.py`, cached per `(courseId, generation_counter)` so re-renders are O(1) until the graph changes.

The Phase 0 implementation in this commit:

- Implements the four load-bearing algorithms — cycle detection, deterministic topological sort, prerequisite walk, effective-difficulty propagation — as pure functions over the agent contracts from §5.
- Implements the **KG expander** that Deliverable 6's retriever calls when reranking is thin.
- Implements the **Cytoscape spec generator** that the FE consumes.
- Mirrors the public types in `packages/knowledge-graph` for the TypeScript side.
- Adds pytest coverage for: acyclic validation, cycle reporting, topological order determinism, prerequisite walks, expansion ranking, and Cytoscape parse round-trip.

## (b) Architecture artifacts

### Algorithms

| Algorithm | Module | Time | Notes |
|---|---|---|---|
| `validate_dag` | `graph.algorithms` | O(V + E) | DFS with tri-colour marking. Returns the first cycle found as a list of concept ids; empty list means acyclic. |
| `topological_order` | `graph.algorithms` | O(V + E + V log V) | Kahn's algorithm with alphabetical tie-break on `concept.id`. Required by Roadmap Planner. |
| `prerequisites_of` | `graph.algorithms` | O(V + E) | BFS along reversed `prerequisite_of` edges. Returns ordered ancestors closest-first. |
| `effective_difficulty` | `graph.algorithms` | O(V + E) | `max(self.difficulty, max(prereq.effective_difficulty))`. Walks once in topo order. |
| `expand_one_hop` | `graph.expander` | O(V + E) | Used by RAG when reranking returns ≤ 2 chunks above the support threshold. Returns concept ids ranked by edge weight and kind priority. |
| `to_cytoscape` | `graph.cytoscape` | O(V + E) | Generates the Cytoscape `elements[]` JSON; embeds `meta.kindPriority` for FE-side filtering. |

### Edge kinds — priority for expansion

The expander walks edges in this priority order until it collects enough candidates:

| Kind | Priority | Direction respected? | Notes |
|---|---|---|---|
| `example_of` | 1 (highest) | Yes (target side) | Concrete examples are the cheapest way to ground a refusal. |
| `related_to` | 2 | Undirected | Symmetric. |
| `prerequisite_of` | 3 | Yes (source side) | Surfacing prerequisites helps the student fill gaps. |
| `derived_from` | 4 | Yes (source side) | "X is derived from Y" expands toward Y. |
| `contradicts` | 5 (lowest) | Undirected | Surfaced only when the original query explicitly references comparison/contrast. |

### Cytoscape JSON shape

The FE consumes the standard Cytoscape `elements[]` form. Each node carries the concept's domain fields under `data`; each edge carries the edge kind and weight. No positions are embedded when the graph has > 200 nodes — the FE runs layout client-side.

```json
{
  "elements": [
    { "data": { "id": "c-1", "label": "Linear Regression", "difficulty": 30,
                "kind": "concept", "blockRefs": [2, 7] } },
    { "data": { "id": "c-2", "label": "Gradient Descent", "difficulty": 55,
                "kind": "concept", "blockRefs": [8] } },
    { "data": { "id": "e-1", "source": "c-1", "target": "c-2",
                "kind": "prerequisite_of", "weight": 1.0,
                "kindPriority": 3 } }
  ],
  "meta": { "nodeCount": 2, "edgeCount": 1, "isLayoutEmbedded": false }
}
```

### Store boundary

```python
class GraphStore(Protocol):
    async def get_course_graph(self, course_id: str) -> tuple[list[Concept], list[ConceptEdge]]: ...
    async def write_course_graph(self, course_id: str, concepts: list[Concept], edges: list[ConceptEdge]) -> None: ...
```

The Phase 0 in-memory implementation keeps a per-course dict; Phase 1 swaps in the Postgres-backed version that uses the `Concept` / `ConceptEdge` tables from Deliverable 3. Application code never reaches into the tables directly.

### Integration points

- **Curriculum Builder agent (§5)** writes via `GraphStore.write_course_graph` after running `validate_dag` and rejecting cycles.
- **Roadmap Planner agent (§5)** reads via `topological_order` and `effective_difficulty`.
- **Retriever (§6)** calls `expand_one_hop` when the `rag.kg-expand` flag is on and reranking is thin.
- **Workspace API (§4)** `GET /v1/courses/{id}/graph` serves the Cytoscape spec, cached in Redis keyed on `(courseId, generation_counter)`.

### Cache invalidation

Every write to `Concept` or `ConceptEdge` bumps a `concept_graph_generation` row counter for the course. The Redis-cached Cytoscape blob and the `kg.expand` results both key on this counter; a write transparently invalidates them.

### Optional Neo4j mirror

When the `kg.neo4j-mirror` flag is on, every Postgres write is replicated to Neo4j via Debezium + a single sink worker. The mirror is read-only and only powers the institution-admin "course analytics" view; the application never queries it for any runtime feature. We can drop the mirror in a single commit if it stops paying its operational cost.

## (c) Trade-offs explicitly rejected

| Rejected | Reason |
|---|---|
| **Neo4j as primary store** | Adds an operational store for a workload Postgres handles trivially at our scale (sub-10k nodes per course). Cycles are detected in O(V + E) without paying Cypher overhead. |
| **Adjacency-list JSONB column on `Course`** | Loses FK integrity, breaks per-tenant RLS, and makes "find all courses citing concept X" a sequential scan. Two normalised tables are simpler. |
| **Letting any agent write edges directly** | Curriculum Builder is the single writer. All other agents read. This is the only way `validate_dag` stays meaningful. |
| **Topological order without deterministic tie-break** | Insertion-order-dependent output makes roadmaps non-reproducible across regenerations. We sort tied nodes alphabetically by `concept.id` (UUIDs are stable). |
| **Cycle reporting via "True / False"** | Useless for debugging. The validator returns the actual cycle path so the Curriculum Builder agent can include it in the error returned to the orchestrator. |
| **Pre-rendering Cytoscape positions for every graph** | Layout cost grows non-linearly. We embed positions only when node count ≤ 200; larger graphs ship topology only and the FE lays them out. |
| **One global graph spanning all courses in a tenant** | Conflates unrelated material and explodes the expander's cost. The graph is per-course; cross-course discovery uses Meilisearch full-text search instead. |
| **Storing edge weights as floats free of constraint** | Wide-open floats invite garbage values that break ranking. Weights are clamped to `[0, 1]` at the Pydantic boundary. |
| **`contradicts` as a default expansion target** | Surfacing contradictory material on every query produces user confusion. Expansion uses `contradicts` only when the query explicitly contains compare/contrast vocabulary. |
| **In-place mutation of the graph during retrieval** | KG expansion is read-only. Any new edges discovered at query time are dropped; persisting them would require a write-path concurrent with read traffic, which we don't need. |
